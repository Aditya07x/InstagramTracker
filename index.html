01
Why, What, How

Not how long. How captured.

Two users spending the same five minutes on Instagram can have completely different experiences. One is engaged. One is captured. Reelio tells them apart — using scroll rhythm, dwell patterns, re-entry gaps, and what you tell it after.

Why this exists

Time-on-app is a blunt instrument. Reelio tracks the texture of a session — friction, drift, exit conflict — so risk feels interpretable, not vague.

What is measured

Every reel writes 100 columns: dwell, velocity, back-scrolls, ambient light, posture, battery state, previous app, and your own regret ratings after.

How to read this page

Scroll to go deeper. Architecture → signals → model logic → dashboards. Each section is one layer of the same system.

Architecture

Five steps. One continuous loop.

From the moment you open Instagram to the score on your dashboard — this is the exact pipeline running on your device.

Step 1
InstaAccessibilityService (Kotlin)
First, the Android service records what actually happened during the session in real time. It logs dwell, speed, pauses, rewatches, exits, return gaps, light, posture, charging state, source app, and timing. Think of this as the raw behavior journal.
AccessibilityService SensorManager Room + CSV append-only logging
Step 2
Schema guard and preprocessing (Python)
Second, Python checks that the session data is complete and trustworthy. It validates required columns, removes duplicate or corrupted rows, normalizes labels, and builds derived features such as intent mismatch and fatigue risk when available.
EXPECTED_SCHEMA_VERSION=5 REQUIRED_COLUMNS enforcement dedupe_session_rows preprocess_session
Step 3
ALSE inference core (ReelioCLSE)
Third, ALSE estimates the hidden session state over time. It combines HMM inference, gap-aware transitions, drift checks, contextual priors, and confidence scoring. This is where doom probability is computed, using a fixed threshold of 0.55.
HMM CTMC regime detector confidence breakdown
Step 4
Heuristic anatomy and alignment loop (DoomScorer)
Fourth, a parallel scorer turns risk into plain reasons you can interpret. It uses seven drivers: session length, exit conflict, rapid re-entry, automaticity, dwell collapse, rewatch compulsion, and environment. Weights are re-aligned against HMM output so the explanation stays consistent with the model state.
7 component anatomy dynamic weights intent mismatch amplification fatigue proxy input
Step 5
Dashboard payload and UI projection
Fifth, run_dashboard_payload packages transitions, confidence, drivers, circadian map, feature weights, and session summaries. The dashboards render this same structure so what you see matches what the model actually produced.
PIPELINE_VERSION=6 model_parameters historical_drivers feature_weights

04
ALSE Internals

A model that learns your normal first.

ALSE runs a 2-state Hidden Markov Model calibrated entirely to you — not population averages. Nine architectural constraints keep it stable, honest about uncertainty, and correctable when it drifts.

01

Personalized Bayesian Baseline

All model priors are anchored to your rolling personal history, not population averages. Your baseline dwell time, scroll speed, and session length are tracked via exponential moving average.

02

Self-Calibrating Emission Weights

After every session, KL divergence is computed per feature between CASUAL and DOOM distributions. Features that separate states better get higher weight. Weights normalize to 1.0 automatically.

03

Hierarchical Temporal Memory

Three EMA memory banks — recent (ρ=0.60), medium (ρ=0.85), and long (ρ=0.97) — are blended to form M-step sufficient statistics. Regime alerts shift weight toward recent memory automatically.

04

Continuous-Time Markov Chain

Session-gap transitions use matrix exponentiation on learned CTMC rates, not a fixed matrix. A 2-minute gap barely shifts state probabilities; an 8-hour sleep gap decays fully toward stationary.

05

Survival Framing

Session stopping is modeled as a geometric hazard rate per state, estimated via Beta posterior from session lengths. Doom sessions have lower hazard — they last longer by design.

06

Regime Change Detector

A CUSUM changepoint detector monitors doom history, dwell patterns, and session timing. When fired, the long-memory bank is frozen so a vacation or illness does not corrupt your baseline.

07

Sparse-Data Guard

Model confidence is gated on data volume and state separation. Before convergence, the model backs off toward priors rather than producing overconfident scores from thin data.

08

Contextual State Priors

A logistic regression on 9 context features (time of day, previous app, day of week, etc.) computes the session-start prior probability of being in a doom state before any reel data arrives.

09

Composite Doom Score

A parallel heuristic DoomScorer runs 7 named components — session length, exit conflict, rapid re-entry, scroll automaticity, dwell collapse, rewatch compulsion, and environment risk — with weights that track the HMM via online learning.

10

Extended Regret Validator

Uses regret labels to correct repeated prediction bias.

11

Confidence Decomposition

Breaks confidence into parts so trust stays transparent and explainable.

12

Dynamic Alignment

Keeps heuristic explanations synchronized with HMM behavior.
